# Migrating from 0.2 to 0.3

fastbff 0.3.0 replaces the validator-driven transformer model with an explicit
**resolve pipeline** and makes the executor **async-native** (ADR 0002). The
public API changed; this guide maps every old construct to its replacement.

## At a glance

| 0.2 | 0.3 |
| --- | --- |
| `@app.transformer` + `BatchArg` + `build_transform_annotated` | `Annotated[T, Resolve(SomeEntityQuery)]` or `Annotated[T, Resolve(resolver=fn)]` |
| `query_executor.fetch(...)` on a **sync** endpoint | inject `SyncQueryExecutor`, `sync_query_executor.fetch(...)` |
| `await query_executor.afetch(...)` on an **async** endpoint | `await query_executor.fetch(...)` |
| `Query[dict[K, V]]` + an iterable field (implicit entity cache) | `EntityQuery[K, V]` with an `ids` field |
| `TransformerAnnotation`, `transformer_metadata`, `validate_batch` | removed â€” no equivalent needed |
| `BatchContextMissingError`, `AsyncDispatchError`, `TransformerRegistrationError` | removed â€” cannot occur; `ResolveRegistrationError` for bad `Resolve` fields |

## 1. Transformers â†’ `Resolve`

A transformer used to be a `@transformer`-decorated function turned into a field
type with `build_transform_annotated`, populating the field as a Pydantic
validator:

```python
# 0.2
@app.transformer
def transform_owner(
    owner_id: int,
    batch: BatchArg[int],
    query_executor: Annotated[QueryExecutor, Depends(QueryExecutor)],
) -> UserDTO | None:
    return query_executor.fetch(FetchUsers(ids=batch.ids)).get(owner_id)

OwnerAnnotated = build_transform_annotated(transform_owner)

class TeamDTO(BaseModel):
    id: int
    owner: OwnerAnnotated
```

In 0.3 the field declares a `Resolve` directly. The raw row value (`owner`, an
id) is the key into the target `EntityQuery`'s `dict[K, V]` result:

```python
# 0.3
class FetchUsers(EntityQuery[int, UserDTO]):
    ids: frozenset[int]

@app.queries
def fetch_users(query: FetchUsers, session: DBSession) -> dict[int, UserDTO]:
    ...

class TeamDTO(BaseModel):
    id: int
    owner: Annotated[UserDTO | None, Resolve(FetchUsers)]   # key = raw row['owner']
```

For custom logic (filtering, deriving keys, calling something other than an
`EntityQuery`), use the **resolver** form â€” a batch-first function that receives
the collected ids and returns `dict[key, value]`:

```python
# 0.3
async def resolve_owner(ids: frozenset[int], executor: QueryExecutor) -> dict[int, UserDTO]:
    users = await executor.fetch(FetchUsers(ids=ids))
    return {i: u for i, u in users.items() if u.active}

class TeamDTO(BaseModel):
    id: int
    owner: Annotated[UserDTO | None, Resolve(resolver=resolve_owner)]
```

The executor is injected into a resolver by **type** (`executor: QueryExecutor`,
no `Depends`); any `Annotated[Dep, Depends(...)]` params are injected exactly as
they were for handlers. Resolvers are discovered automatically from the response
models your queries return â€” there is no `@resolver` decorator to add.

## 2. Endpoints: sync vs async

`QueryExecutor.fetch` is now a coroutine.

```python
# 0.3 â€” async endpoint
@fastapi_app.get('/teams')
async def list_teams(
    query_executor: Annotated[QueryExecutor, Depends(QueryExecutor)],
) -> list[TeamDTO]:
    return await query_executor.fetch(FetchTeams())

# 0.3 â€” sync endpoint
@fastapi_app.get('/teams')
def list_teams(
    sync_query_executor: Annotated[SyncQueryExecutor, Depends(SyncQueryExecutor)],
) -> list[TeamDTO]:
    return sync_query_executor.fetch(FetchTeams())
```

`SyncQueryExecutor.fetch` bridges onto the event loop (Starlette runs the sync
endpoint in a worker thread), so async handlers still work from a sync endpoint.
`afetch` is gone â€” async endpoints use `await ...fetch(...)`.

## 3. Entity caching is opt-in

Entity-level caching (overlapping id sets share cached entries, only missing ids
are fetched) used to trigger for any `dict[K, V]`-returning query that happened
to have an iterable field. Now it is explicit:

```python
# 0.2
class FetchUsers(Query[dict[int, UserDTO]]):
    ids: frozenset[int]

# 0.3
class FetchUsers(EntityQuery[int, UserDTO]):
    ids: frozenset[int]
```

A plain `Query[dict[K, V]]` still works but gets **call-level** caching only, so
adding an unrelated iterable field can never silently change cache semantics.

## 4. Removed exceptions

`BatchContextMissingError` (a model validated outside a dispatch boundary),
`AsyncDispatchError` (an async handler reached on a non-awaitable path), and
`TransformerRegistrationError` no longer exist â€” the situations that produced
them are structurally impossible in the resolve pipeline. Mis-declared `Resolve`
fields (e.g. `Resolve(QueryType)` pointing at a non-`EntityQuery`, or both a
query type and a resolver) raise `ResolveRegistrationError` at finalize/mount.

## 5. Standalone model validation

DTOs are now plain models: `TeamDTO.model_validate(row)` works anywhere once the
resolved values are present, with no validation context required. Composition
happens in the executor's render phase, not inside `model_validate`.
